SKILL: db_contract_script_runner
Execute pre-approved contract SQL/shell scripts with SHA-256 checksum verification, approval gating, automatic rollback on failure, and full structured audit logging.
risk: highL3requires_approvalv2.1.0DBA only
8-Step Execution Protocol
1
Validate Inputs
Assert script_name, target_database in registry, approval_ticket_id valid for production targets.
↓
2
Fetch Script + Verify SHA-256
Download from S3 skills/.../contract_scripts/. Hash must match L1 manifest. Mismatch → security incident, hard abort.
↓
3
Pre-flight Connection Check
SELECT 1 with 5s timeout. Abort with CONNECTION_FAILED if unreachable.
↓
4
Dry-run (if requested)
Wrap in BEGIN/ROLLBACK. Capture row counts, lock types, DDL changes. Return impact report and exit.
↓
5
Execute in Transaction
BEGIN with statement_timeout and lock_timeout = 10s. On error → ROLLBACK → execute paired rollback script.
↓
6
Post-execution Validation
Run *_validate.sql from S3. Assert result->>'passed' = 'true'.
↓
7
Audit Log
Write structured JSON record: role, skill, script, timestamp, outcome, rows_affected, checksum_verified.
↓
8
Notify via Shared Skill
Delegates to notification_dispatch (shared skill). Posts summary to #dba-alerts.

SKILL: db_schema_migrate
Apply a versioned schema migration. Enforces sequential version ordering, idempotency, and rollback support. Uses pre-authored migration scripts from S3.
risk: highL2requires_approvalidempotent
SQLVersion guard + idempotency + transaction
-- Step 1: Sequence guard — no version gaps allowed
SELECT version FROM schema_version ORDER BY applied_at DESC LIMIT 1;
-- current=3.3.2, requested=3.4.0 → ALLOWED
-- current=3.3.2, requested=3.5.0 → REJECTED (gap detected)

-- Step 2: Idempotency — never re-apply
SELECT 1 FROM schema_version WHERE version = '3.4.0';
-- Exists → return no_op (success, no execution)

-- Step 3: Execute + record in single transaction
BEGIN;
  -- [migration DDL injected from S3 script]
  INSERT INTO schema_version (version, applied_at, applied_by, checksum)
  VALUES ('3.4.0', NOW(), 'dba_agent', '<sha256>');
COMMIT;

SKILL: db_health_report
Read-only health assessment: bloat, unused indexes, replication lag, lock contention, connection pool saturation, slow queries. Green/Yellow/Red thresholds.
risk: lowL2read-onlyv1.4.0
SQLBloat detection + connection saturation
-- Table bloat analysis
SELECT tablename,
  pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) total_size,
  ROUND(100*(n_dead_tup::numeric/NULLIF(n_live_tup+n_dead_tup,0)),2) dead_pct
FROM pg_stat_user_tables
ORDER BY dead_pct DESC LIMIT 20;

-- Connection pool saturation
SELECT
  (SELECT setting::int FROM pg_settings WHERE name='max_connections') max_conn,
  COUNT(*) used_conn,
  ROUND(100.0*COUNT(*)/(SELECT setting::int
    FROM pg_settings WHERE name='max_connections'), 1) utilisation_pct
FROM pg_stat_activity;

Contract Script: add_indexes_orders_v1.sql
Adds 4 concurrent non-locking indexes to orders schema. Resolves slow query findings (mean 2,340ms) from db_health_report. Paired with rollback + validate scripts.
V3.4.0CHANGE-2024-0847CONCURRENT — no table lock
SQLadd_indexes_orders_v1.sql · v1.0.0
SET statement_timeout = '0';  -- Disabled for CONCURRENT builds
SET lock_timeout      = '5s';
SET work_mem          = '256MB';

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_orders_customer_status
ON orders (customer_id, status)
WHERE status NOT IN ('archived', 'cancelled');

SELECT pg_sleep(2); -- I/O pressure relief between builds

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_orders_created_at_status
ON orders (created_at DESC, status) INCLUDE (customer_id, total_amount);

SELECT pg_sleep(2);

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_order_items_order_id
ON order_items (order_id) INCLUDE (product_id, quantity, unit_price);

SELECT pg_sleep(2);

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_order_items_product_agg
ON order_items (product_id, created_at DESC) INCLUDE (quantity, unit_price);

ANALYZE orders; ANALYZE order_items;

INSERT INTO schema_version (version, description, applied_at, applied_by)
VALUES ('3.4.0', 'add_indexes_orders_v1', NOW(), current_user)
ON CONFLICT (version) DO NOTHING;

Rollback: add_indexes_orders_v1_rollback.sql
Paired rollback — drops all 4 indexes in reverse order. IF EXISTS guards make it safe to run against a partially applied migration.

SQLadd_indexes_orders_v1_rollback.sql
SET lock_timeout = '5s';
DROP INDEX CONCURRENTLY IF EXISTS idx_order_items_product_agg;  SELECT pg_sleep(1);

Validation: add_indexes_orders_v1_validate.sql
Returns a single JSON row. Orchestrator checks result->>'passed' = 'true' before marking the execution as successful.
returns JSONorchestrator-parsed
SQLadd_indexes_orders_v1_validate.sql
WITH idx AS (
  SELECT indexname,
    CASE WHEN indisvalid THEN 'valid' ELSE 'INVALID' END status
  FROM pg_indexes JOIN pg_class ON relname=indexname
  JOIN pg_index ON indexrelid=pg_class.oid
  WHERE indexname IN (
    'idx_orders_customer_status', 'idx_orders_created_at_status',
    'idx_order_items_order_id',    'idx_order_items_product_agg'
  )
)
SELECT json_build_object(
  'passed',        (SELECT COUNT(*) = 4 AND bool_and(status='valid') FROM idx),
  'indexes_valid', (SELECT COUNT(*) FROM idx WHERE status='valid'),
  'errors',        (SELECT COALESCE(json_agg(indexname||': '||status),'[]'::json)
                    FROM idx WHERE status != 'valid')
) AS result;

-- Orphan record detection (order_items without parent orders)
SELECT oi.id, oi.order_id
FROM order_items oi
LEFT JOIN orders o ON o.id = oi.order_id
WHERE o.id IS NULL;   -- Should return 0 rows

-- Referential integrity — orders without valid customers
SELECT o.id, o.customer_id
FROM orders o
LEFT JOIN customers c ON c.id = o.customer_id
WHERE c.id IS NULL;

-- Null constraint violations in required fields
SELECT COUNT(*) violations
FROM orders
WHERE status IS NULL
   OR total_amount IS NULL
   OR created_at IS NULL;

-- Seeded data consistency check
SELECT fixture_name, expected_count, COUNT(*) actual_count,
  COUNT(*) = expected_count AS matches
FROM qa_fixture_manifest
JOIN orders ON source_fixture = fixture_name
GROUP BY fixture_name, expected_count;

QA Contract Script: regression_suite_v2.sh
Shell contract script that executes the full regression suite using pytest. Parameterised by environment, suite tags, and parallel workers. Outputs JUnit XML + coverage report.
QA contractCHANGE-QA-2024-0112no approval needed
BASHregression_suite_v2.sh · v2.0.0
#!/usr/bin/env bash
# ============================================================
# QA CONTRACT SCRIPT: regression_suite_v2.sh
# Version: 2.0.0   |   Risk: medium
# Checksum: d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3
# ============================================================
set -euo pipefail

# Parameters injected by orchestrator via env vars
TARGET_ENV="${TARGET_ENV:?required}"       # staging | sandbox
RELEASE_TAG="${RELEASE_TAG:?required}"     # e.g. v2.4.1-rc1
SUITE_TAGS="${SUITE_TAGS:-}"               # e.g. "checkout or auth"
WORKERS="${PARALLEL_WORKERS:-4}"
RETRY_COUNT="${FLAKY_RETRY_COUNT:-2}"
OUTPUT_DIR="${OUTPUT_DIR:-/tmp/qa_results}"

# Hard guard: never run against production
if [[ "$TARGET_ENV" == "production" ]]; then
  echo "SECURITY: production execution denied" >&2
  exit 1
fi

echo "▶ Regression Suite · ${RELEASE_TAG} → ${TARGET_ENV}"
mkdir -p "$OUTPUT_DIR"

# Step 1: Seed test fixtures
echo "→ Seeding fixtures..."
python manage.py load_qa_fixtures \
  --env "$TARGET_ENV" \
  --manifest s3://company-skill-library-prod/fixtures/standard_v3.json

# Step 2: Execute test suite
echo "→ Running tests (workers=${WORKERS}, retry=${RETRY_COUNT})..."
pytest tests/ \
  --numprocesses="$WORKERS" \
  --reruns="$RETRY_COUNT" \
  --reruns-delay=2 \
  -m "${SUITE_TAGS:-not slow}" \
  --junit-xml="${OUTPUT_DIR}/results.xml" \
  --cov=src \
  --cov-report=json:"${OUTPUT_DIR}/coverage.json" \
  --cov-report=html:"${OUTPUT_DIR}/coverage_html" \
  --tb=short \
  -v

EXIT_CODE=$?

# Step 3: Upload results to S3
echo "→ Uploading results to S3..."
aws s3 sync "$OUTPUT_DIR" \
  "s3://report-output-bucket/qa/${RELEASE_TAG}/${TARGET_ENV}/"

# Step 4: Teardown fixtures
echo "→ Tearing down fixtures..."
python manage.py teardown_qa_fixtures --env "$TARGET_ENV"

echo "✓ Suite complete. Exit: ${EXIT_CODE}"
exit $EXIT_CODE

QA Contract Script: data_integrity_check.sql
Runs against sandbox/staging after fixture seeding to confirm data consistency before test execution. Returns structured JSON; orchestrator asserts all checks passed.
QA contractread-onlysandbox + staging only
SQLdata_integrity_check.sql · v1.2.0
-- ============================================================
-- QA CONTRACT SCRIPT: data_integrity_check.sql
-- Runs AFTER fixture seeding, BEFORE test execution
-- Returns JSON: { "passed": bool, "checks": [...] }
-- ============================================================

WITH
orphan_items AS (
  SELECT COUNT(*) cnt FROM order_items oi
  LEFT JOIN orders o ON o.id = oi.order_id WHERE o.id IS NULL
),
orphan_orders AS (
  SELECT COUNT(*) cnt FROM orders o
  LEFT JOIN customers c ON c.id = o.customer_id WHERE c.id IS NULL
),
null_violations AS (
  SELECT COUNT(*) cnt FROM orders
  WHERE status IS NULL OR total_amount IS NULL OR created_at IS NULL
),
fixture_counts AS (
  SELECT
    (SELECT COUNT(*) FROM customers)  expected_customers,
    (SELECT COUNT(*) FROM orders)     expected_orders,
    (SELECT COUNT(*) FROM products)   expected_products
)
SELECT json_build_object(
  'passed', (
    (SELECT cnt = 0 FROM orphan_items) AND
    (SELECT cnt = 0 FROM orphan_orders) AND
    (SELECT cnt = 0 FROM null_violations) AND
    (SELECT expected_customers >= 100 FROM fixture_counts)
  ),
  'checks', json_build_array(
    json_build_object('name', 'orphan_order_items',
      'passed', (SELECT cnt = 0 FROM orphan_items),
      'violations', (SELECT cnt FROM orphan_items)),
    json_build_object('name', 'orphan_orders',
      'passed', (SELECT cnt = 0 FROM orphan_orders),
      'violations', (SELECT cnt FROM orphan_orders)),
    json_build_object('name', 'null_constraint_violations',
      'passed', (SELECT cnt = 0 FROM null_violations),
      'violations', (SELECT cnt FROM null_violations)),
    json_build_object('name', 'fixture_minimum_counts',
      'passed', (SELECT expected_customers >= 100 FROM fixture_counts),
      'counts', (SELECT row_to_json(fixture_counts) FROM fixture_counts))
  )
) AS result;
